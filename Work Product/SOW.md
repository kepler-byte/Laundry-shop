## วัตถุประสงค์การจัดทำ Statement of Work (SOW)

เอกสาร Statement of Work (SOW) ฉบับนี้จัดทำขึ้นเพื่อกำหนดขอบเขตการดำเนินงาน เงื่อนไข และข้อตกลงในการพัฒนาระบบบริหารจัดการร้านรับซักรีดเสื้อผ้า ระหว่างผู้ว่าจ้างและผู้รับจ้าง เพื่อให้การพัฒนาและนำระบบไปใช้งานเป็นไปอย่างมีประสิทธิภาพ เป็นไปตามความต้องการของธุรกิจ และสามารถสนับสนุนการดำเนินงานของร้านซักรีดได้อย่างครบถ้วน ถูกต้อง และตรวจสอบได้

## 3. ขอบเขตของโครงการ

### 3.1 ขอบเขตการดำเนินงาน

ระบบงานหลักประกอบด้วยโมดูลต่าง ๆ ดังตารางต่อไปนี้

| โมดูล | ฟังก์ชันการทำงาน | ผู้ใช้งาน |
|------|------------------|-----------|
| 1. ระบบจัดการลูกค้าและสมาชิก | - จัดเก็บและบริหารข้อมูลลูกค้า<br>- จัดการสมาชิก ระดับสมาชิก และสิทธิประโยชน์ | - พนักงานหน้าร้าน<br>- ผู้จัดการ |
| 2. ระบบรับเสื้อผ้า | - บันทึกการรับเสื้อผ้าและรายละเอียด<br>- ระบุประเภทบริการและกำหนดเวลาส่ง | - พนักงานหน้าร้าน |
| 3. ระบบจัดการคำสั่งงาน | - สร้างและจัดลำดับคำสั่งงานซัก/รีด/พับ<br>- มอบหมายงานให้พนักงาน | - พนักงาน<br>- ผู้จัดการ |
| 4. ระบบติดตามสถานะ | - ติดตามสถานะงานแบบเรียลไทม์<br>- แจ้งเตือนความคืบหน้าให้ลูกค้า | - ลูกค้า<br>- พนักงาน |
| 5. ระบบจัดการกระบวนการซัก/รีด | - ควบคุมกระบวนการซัก รีด พับ<br>- บันทึกผลการดำเนินงานแต่ละขั้นตอน | - พนักงานแผนกซัก/รีด |
| 6. ระบบคืนเสื้อผ้า | - ตรวจสอบความเรียบร้อยก่อนคืน<br>- บันทึกการส่งมอบเสื้อผ้า | - พนักงานหน้าร้าน |
| 7. ระบบคำนวณราคาและชำระเงิน | - คำนวณค่าบริการ ส่วนลด และภาษี<br>- รองรับการชำระเงินหลายช่องทาง | - ลูกค้า<br>- ฝ่ายการเงิน |
| 8. ระบบจัดการบริการพิเศษ | - จัดการบริการด่วนและบริการรับ-ส่ง<br>- บริการสำหรับลูกค้าองค์กร | - ผู้จัดการ |
| 9. ระบบสต็อกและอุปกรณ์ | - จัดการน้ำยา วัสดุ และอุปกรณ์<br>- แจ้งเตือนสต็อกใกล้หมดและบำรุงรักษา | - ผู้จัดการ |
| 10. ระบบออกรายงานและ Dashboard | - รายงานการดำเนินงาน การเงิน และสต็อก<br>- Dashboard สำหรับผู้บริหาร | - ผู้บริหาร |

## 4. ข้อตกลงร่วมกัน

การรับประกันระบบ 
- ระยะเวลา 99 day  
- แก้ไขข้อผิดพลาดของระบบ (Bug) ที่เกิดจากการพัฒนา
 
การสำรองข้อมูล (Backup)
- มีการสำรองข้อมูลเป็นระยะ
- ป้องกันข้อมูลสูญหายจากความผิดพลาดของระบบ

ระดับการให้บริการ (SLA) 
- ระบบสามารถใช้งานได้อย่างต่อเนื่อง 
- ระบบตอบสนองการทำงานได้รวดเร็วและเสถียร  

การสนับสนุนและดูแล  
- ให้คำปรึกษาและช่วยแก้ไขปัญหาการใช้งานระบบ 
- ดูแลและแนะนำการใช้งานเบื้องต้นแก่ผู้ใช้งาน

เงื่อนไขเพิ่มเติม 
- การแก้ไขเพิ่มเติมนอกเหนือจากข้อตกลง อาจมีค่าใช้จ่ายเพิ่มเติม
- ข้อตกลงสามารถเปลี่ยนแปลงได้ตามความเหมาะสม

การปรับปรุงระบบในอนาคต (Optional) 
- สามารถปรับปรุงหรือเพิ่มฟังก์ชันใหม่ในอนาคต
- การเปลี่ยนแปลงต้องได้รับความเห็นชอบจากทั้งสองฝ่าย

## 5. การบริหารการเปลี่ยนแปลง (Change Management)

เพื่อให้โครงการดำเนินไปได้อย่างราบรื่นและสามารถควบคุมขอบเขตงาน (Scope), ระยะเวลา (Timeline) และต้นทุน (Cost) ได้อย่างมีประสิทธิภาพ หากมีความต้องการเปลี่ยนแปลงใดๆ เกิดขึ้นนอกเหนือจากที่ระบุไว้ในเอกสารข้อกำหนดความต้องการ (SRS) หรือสัญญาฉบับนี้ ทั้งสองฝ่ายตกลงที่จะปฏิบัติตามกระบวนการดังนี้:

### 5.1 นิยามของการเปลี่ยนแปลง (Definition of Change)
การเปลี่ยนแปลงที่ต้องผ่านกระบวนการนี้ ครอบคลุมถึง:
* การเพิ่มฟังก์ชันการทำงานใหม่ (New Features)
* การแก้ไขฟังก์ชันเดิมที่ได้รับอนุมัติไปแล้ว (Modification of approved features)
* การเปลี่ยนแปลง Business Logic หรือ Workflow ที่ส่งผลกระทบต่อโครงสร้างระบบ
* การเปลี่ยนแปลงโครงสร้างฐานข้อมูล หรือ Design ที่ได้ข้อสรุปไปแล้ว

### 5.2 ขั้นตอนการดำเนินงาน (Change Control Process)

| ขั้นตอน | กิจกรรม | ผู้รับผิดชอบ | ระยะเวลา |
| :--- | :--- | :--- | :--- |
| **1. ยื่นคำขอ** | ผู้ว่าจ้างจัดทำเอกสาร **"แบบฟอร์มขอเปลี่ยนแปลง" (Change Request Form - CRF)** ระบุรายละเอียดสิ่งที่ต้องการเปลี่ยนแปลงและเหตุผล | ผู้ว่าจ้าง | - |
| **2. ประเมินผลกระทบ** | ทีมพัฒนาทำการวิเคราะห์ผลกระทบ (Impact Analysis) ด้านเทคนิค, ระยะเวลา, และค่าใช้จ่ายที่อาจเพิ่มขึ้น | ผู้รับจ้าง | 1-3 วันทำการ |
| **3. เสนอแนวทาง** | ผู้รับจ้างส่งกลับ **"ข้อเสนอการเปลี่ยนแปลง" (Change Proposal)** ซึ่งระบุจำนวนวันทำงาน (Man-days) และค่าใช้จ่ายเพิ่มเติม (ถ้ามี) ให้ผู้ว่าจ้างพิจารณา | ผู้รับจ้าง | - |
| **4. การอนุมัติ** | ผู้ว่าจ้างพิจารณาข้อเสนอ <br> - หาก **"อนุมัติ"**: ลงนามยืนยันเพื่อเริ่มดำเนินการ <br> - หาก **"ไม่อนุมัติ"**: ยึดตามขอบเขตงานเดิม | ผู้ว่าจ้าง | ภายใน 3 วันทำการ |
| **5. ดำเนินการ** | ทีมพัฒนาเริ่มดำเนินการแก้ไขตามที่ตกลง และปรับปรุงแผนงานโครงการ (Project Plan) ใหม่ | ผู้รับจ้าง | ตามที่ตกลง |

### 5.3 อัตราค่าบริการเพิ่มเติม (Service Rate for Changes)
หากการเปลี่ยนแปลงส่งผลให้ต้องมีค่าใช้จ่ายเพิ่มเติม จะใช้วิธีการคำนวณดังนี้:
1.  **กรณีงานเล็กน้อย (Minor Change):** หากใช้เวลาแก้ไขไม่เกิน 4 ชั่วโมงการทำงาน จะ **ไม่คิดค่าใช้จ่าย** (จำกัดไม่เกิน 3 ครั้งตลอดโครงการ)
2.  **กรณีงานเพิ่มเติม (Major Change):** คิดค่าบริการตามระยะเวลาที่ใช้จริง (Man-days) ในอัตรา [ระบุราคา] บาท/วัน (หรือตามที่ตกลงกันในสัญญาแนบท้าย)
3.  **ผลกระทบต่อระยะเวลา:** จำนวนวันที่ใช้ในการเปลี่ยนแปลง จะถูกนำไปบวกเพิ่มในระยะเวลาส่งมอบงานงวดนั้นๆ โดยไม่ถือเป็นความล่าช้าของผู้รับจ้าง

### 5.4 คณะทำงานพิจารณาการเปลี่ยนแปลง (Change Control Board - CCB)
สำหรับการเปลี่ยนแปลงที่มีผลกระทบสูง (เช่น กระทบระยะเวลาเกิน 2 สัปดาห์ หรือกระทบงบประมาณเกิน 10%) จะต้องได้รับการอนุมัติจากคณะทำงาน CCB ซึ่งประกอบด้วย:
* ตัวแทนผู้มีอำนาจตัดสินใจจากฝั่งผู้ว่าจ้าง 1 ท่าน
* Project Manager จากฝั่งผู้รับจ้าง 1 ท่าน
